DESIGN SUMMARY
================================================================================

PRINCIPLE: Minimize complexity. Store responses as immutable documents.

SCHEMA (3 tables):
  • companies      - static config (company_id, name, slug, question_count)
  • questions      - static templates (company_id, question_id, title)
  • responses      - form submissions (company_id, name, email, ..., questions_json)

KEY DECISIONS:
  ✓ One response row = entire submission (no multi-row inserts)
  ✓ questions_json = array of {id, score} (no separate question_responses table)
  ✓ Immutable writes = only inserts, no updates (audit trail)
  ✓ Indexed on (company_id, created_at) = O(log n) queries
  ✓ Maturity level denormalized = instant filtering without JOIN

ENDPOINTS (3 total):
  ✓ POST /api/responses           - Submit form (public, no auth)
  ✓ GET /api/responses            - List with filters (requires API_SECRET)
  ✓ GET /api/responses/:id        - Get detail (requires API_SECRET)
  ✓ GET /api/responses/stats/:co  - Aggregate stats (requires API_SECRET)

VALIDATION HIDDEN IN WORKER:
  • Email format, question count, score ranges
  • Duplicate email detection (24h window)
  • Company existence check
  • Missing field detection

KEY FEATURES
================================================================================

✓ Single endpoint for form ingestion (no multi-step process)
✓ JSON response storage (no relational normalization)
✓ Indexed filtering (company_id, email, maturity_level)
✓ Pagination (50 results per page, configurable)
✓ Bearer token authentication (API_SECRET)
✓ Duplicate detection (24h window per email)

================================================================================
COMPLEXITY HIDDEN FROM SCHEMA
================================================================================

✗ Email validation          → Worker middleware
✗ Question count validation → Worker validation
✗ Score range checking      → Worker validation
✗ Maturity level calculation → Worker (calculated from score)
✗ Duplicate detection       → 24h window query + app logic
✗ Authentication           → Bearer token in headers
✗ Pagination               → App-layer cursor (offset/limit)
✗ Filtering                → WHERE clauses built dynamically
✗ Aggregation              → GROUP BY queries
✗ Question title enrichment → Separate SELECT on join

RESULT: Schema is pure document store. All logic lives in application.

SCALE TARGETS
================================================================================

Current design handles:
  • < 100k responses per company ✓
  • < 1M total responses ✓
  • < 10 questions per company ✓
  • < 100 companies ✓

If you exceed these:
  • Index on (company_id, created_at) still performs
  • Create stats cache (nightly aggregation)
  • Archive old responses to R2 (cold storage)
  • Partition responses by year

================================================================================
DEPENDENCY STRATEGY
================================================================================

COMPANY CONFIGS:
  • Pre-loaded from D1 companies table
  • Verified on every submission
  • Change = insert new row (immutable)

QUESTIONS:
  • Pre-loaded from D1 questions table
  • Referenced by question_id in submissions
  • Change = insert new question_id (versioning)

NO EXTERNAL DEPENDENCIES:
  • No external API calls
  • No cache invalidation needed
  • No sync layer required
  • Single source of truth = D1
